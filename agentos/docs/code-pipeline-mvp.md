## Инструкция по прохождению кодового конвейера (MVP Flow)

Этот документ описывает минимальный жизнеспособный процесс (MVP) для внесения кодовых изменений. До момента внедрения автоматических валидаторов весь процесс опирается на контракт, ручную сверку и явное одобрение человека.

### 1. Входные данные (Inputs)
- Утвержденный **Task Brief** со статусом `READY_FOR_EXECUTION`.
- Четко определенный **Risk Profile** (одобренный человеком).
- Списки **Allowed Changes** и **Forbidden Changes**.

### 2. Процесс исполнения (Execution)
Агент приступает к изменению кода. Во время работы он соблюдает следующие ограничения:
1. Писать код только в рамках файлов/директорий из `Allowed Changes`.
2. Остановиться и перейти в статус `UNKNOWN_BLOCKED`, если возникает необходимость изменить неразрешенные файлы.
3. Собирать доказательства изменений (Evidence).

### 3. Отчетность и проверка (Reporting & Review)
По завершению редактирования файлов агент **обязан**:
1. Сгенерировать **Execution Report** (используя шаблон `execution-report.md`), где он описывает, что именно было сделано. Статус отчета: `READY_FOR_REVIEW`.
2. Сгенерировать **Evidence Report** (используя шаблон `evidence-report.md`). Доказательства должны включать:
   - Список затронутых путей.
   - Diff изменений.
   - Вывод валидатора (на этапе MVP статус валидатора всегда `NOT_RUN`).
3. Прекратить работу и ожидать **Human Review**.

### 4. Human Review (Ручное одобрение)
Человек анализирует предоставленные отчеты.
- Если человек одобряет работу, он генерирует **Human Review** документ (на базе шаблона) со статусом `APPROVED`. Только после этого изменения могут быть вмержены или признаны завершенными.
- `PASS ≠ approval`. Отчет агента об успешной работе не означает, что код принят.
