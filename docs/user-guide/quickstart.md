# AOS-FARM Quickstart: Быстрый старт для новых пользователей

## 1. Что такое AOS-FARM?
AOS-FARM — это защищенная среда (репозиторий) для создания, тестирования и контроля автономных ИИ-агентов. Важно понимать, что система не претендует на готовность к production, а автономное выполнение (runner) сейчас отключено ради безопасности. Вы контролируете каждый шаг ИИ.

## 2. Что открыть в первую очередь
Если вы только что открыли репозиторий, не пытайтесь сразу менять код. Ознакомьтесь со следующими файлами:
* `README.md` (если присутствует)
* `docs/user-guide/project-map.md` (карта проекта)
* `docs/user-guide/glossary.md` (словарь терминов)
* `docs/user-guide/first-consumer-workflow.md` (руководство первого потребителя)

## 3. Иерархия источников (Source Hierarchy)
Этот Quickstart — точка входа для пользователя, но **не высший авторитет**.
* Канонические файлы `00_AOS_Core_Control.md`, `01_AOS_Assembly_Pipelines_and_Build_Roadmap.md` и `02_AOS_Governance_Control_Module_and_Safety_Rules.md` (называемые 00/01/02) являются высшим источником истины.
* Если этот Quickstart противоречит 00/01/02, следуйте правилам из 00/01/02.
* Если конфликт непонятен, остановитесь и установите статус `HUMAN_REVIEW_REQUIRED`.

## 4. Минимальный безопасный путь для старта
1. Прочтите карту проекта (project map).
2. Прочтите словарь терминов (glossary).
3. Прочтите руководство первого потребителя (first consumer workflow).
4. Выберите **небольшую первую задачу** (small first task).
5. Заполните шаблон задания на контролируемую задачу (controlled task brief).
6. Попросите агента подготовить ограниченный план (bounded plan).
7. Оцените и назначьте профиль риска (Risk Profile) самостоятельно, как человек.
8. Авторизуйте выполнение (execution authorization) **только после проверки** плана.
9. Проверьте доказательства (Evidence) работы агента.
10. Авторизуйте коммит (commit authorization) **только после проверки** отчетов.
11. Авторизуйте отправку (push authorization) **только после создания коммита**.

## 5. Что такое "Небольшая первая задача"?
Это задача, которая:
* Не изменяет 00_AOS_Core_Control.md, 01_AOS_Assembly_Pipelines_and_Build_Roadmap.md или 02_AOS_Governance_Control_Module_and_Safety_Rules.md.
* Не требует выпуска релиза.
* Не требует использования в production.
* Не требует включения автономного runner'а.
* Не требует изменений в CI-пайплайнах.
* Не требует базы данных, RAG или векторного хранилища.
* Полностью помещается в один документ Task Brief.
* Имеет четкий список разрешенных файлов и список запрещенных действий.
* Легко проверяется человеком через финальный отчет.

**Примеры подходящих задач:**
* Добавить страницу словаря терминов для пользователей.
* Добавить небольшой шаблон чек-листа.
* Добавить короткую страницу руководства.
* Уточнить одну не каноническую страницу документации.

**Примеры НЕ подходящих задач (слишком больших):**
* Изменять канонические источники 00/01/02.
* Менять процесс релиза.
* Включать автономный runner.
* Добавлять CI пайплайны.
* Заявлять о готовности к production.
* Реструктурировать репозиторий.

## 6. Что такое Bounded Prompt (Ограниченный промпт)?
Ограниченный промпт четко описывает ИИ-агенту рамки дозволенного. Он включает:
* Точную цель.
* Разрешенные файлы.
* Запрещенные файлы.
* Запрещенные действия (destructive operations).
* Обязательные источники для чтения.
* Правила для профиля риска.
* Требования к верификации и формату отчета.
* Границы для коммита/пуша/релиза (запрет самостоятельных действий).

**Пример ограниченного промпта:**
*(This is an example, not a universal script. Adapt it to the actual task while preserving safety boundaries.)*
> Создай файл `docs/user-guide/example.md` с описанием базовой концепции. Не изменяй `00`, `01`, `02` или README. Не делай commit, не делай push, не создавай release. Создай отчеты `implementation-report.md` и `commit-authorization-package.md`, а также DRAFT чекпойнт для авторизации. Если что-то неясно — остановись (HUMAN_REVIEW_REQUIRED). Профиль риска: MEDIUM_RISK_GUIDED.

## 7. Чего пользователь делать НЕ ДОЛЖЕН
* Не разрешайте ИИ-агенту действовать полностью автономно (runner).
* Не позволяйте ИИ-агенту самому себя одобрять.
* Не позволяйте ИИ-агенту самостоятельно назначать себе статус `LOW_RISK_FAST`.
* Не считайте успешное завершение скрипта (PASS) вашим одобрением (approval).
* Не считайте наличие доказательств (Evidence) автоматическим одобрением.
* Не считайте успешный проход тестов (CI PASS) одобрением релиза.
* Не считайте статус `UNKNOWN` приемлемым (OK).
* Не считайте статус `NOT_RUN` успешным (PASS).
* Не одобряйте не глядя изменения в защищенных/канонических файлах.
* Не разрешайте коммит, пуш или релиз без создания файла Human Authorization.

## 8. Практический путь "Первые 10 минут"
1. Откройте репозиторий.
2. Бегло просмотрите структуру (папки `docs`, `reports`, `templates`).
3. Прочитайте `docs/user-guide/quickstart.md` (вы здесь).
4. Прочитайте словарь `docs/user-guide/glossary.md`.

## 9. Практический путь "Первая контролируемая задача"
1. Скопируйте шаблон `templates/first-controlled-task-brief-template.md`.
2. Заполните его своей небольшой задачей (например, "Добавить файл с описанием").
3. Передайте этот бриф агенту.

## 10. Что отправлять ИИ-агенту (What to send to the AI agent)
* Явное указание на вашу задачу (Bounded Prompt).
* Инструкцию прочитать канонические источники (00, 01, 02).
* Список файлов, которые он может создавать или менять.
* Запрет на коммиты и пуши без вашего явного чейкпойнта.

## 11. Что требовать от ИИ-агента на проверку (What to bring back to reviewer)
* Фактические изменения (новые/измененные файлы).
* Отчет о реализации (Implementation Report).
* Отчет о проверке границ (Verification Report).
* Пакет авторизации коммита и DRAFT-чекпойнт (DRAFT Human Checkpoint), ожидающий вашего решения.

## 12. Когда нужно останавливаться (When to stop fail-closed)
Остановите работу и установите статус `BLOCKED` (или `HUMAN_REVIEW_REQUIRED`), если:
* Агент пытается сам одобрить свои изменения.
* Агент пытается сделать коммит, пуш или релиз без Human Authorization.
* Агент изменяет или предлагает изменить 00/01/02 без создания защищенного чекпойнта.
* Агент считает статус PASS за ваше одобрение.
* Агент считает файлы Evidence за ваше одобрение.
* Агент воспринимает CI PASS как одобрение публикации.
* Появляется статус UNKNOWN, но агент продолжает работать.
* Статус NOT_RUN выдается агентом за PASS.
* Появляются признаки работы автономного runner'а.
* Агент заявляет готовность к production use.
* Агент пытается создать GitHub релиз без отдельного разрешения.

## 13. Явные границы системы (Clear statement)
* AOS-FARM **не претендует** на использование в production.
* Релиз GitHub **не создается**, если это не разрешено отдельным чекпойнтом.
* Автономное выполнение агента (runner) **не является частью** текущего безопасного пути.
* Этот Quickstart **не авторизует** выполнение кода, коммиты, пуши, релизы, изменения в production или модификацию канонических источников.
